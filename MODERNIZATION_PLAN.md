# План модернизации проекта `achkasov_dev`

## 1. Цели модернизации

- Повысить надежность и предсказуемость сборки/деплоя.
- Устранить архитектурные рассогласования между backend и frontend.
- Повысить безопасность (аутентификация, секреты, CORS, хранение токенов).
- Повысить качество кода и покрытие тестами.
- Подготовить проект к production-эксплуатации.

## 2. Ключевые проблемы (текущее состояние)

### Backend
- Конфигурация частично хардкодится в коде (включая security-чувствительные значения).
- Есть рассогласование DI и реализаций репозиториев (admin-слой).
- CRUD-логика местами не использует корректный поиск по первичному ключу.
- Ошибки/опечатки в моделях и потенциально конфликтная схема БД.
- Недостаточное покрытие тестами.

### Frontend
- Сборка не проходит из-за отсутствующих модулей (`lib/api`, `lib/utils`).
- Часть API URL зашита строками вместо централизованной конфигурации.
- Авторизация админки хранит флаг доступа в `localStorage` без полноценной серверной валидации сессии.

### Platform / DevOps
- Документация и реальная конфигурация контейнеров местами рассинхронизированы.
- Не хватает CI-пайплайна с обязательными проверками.

## 3. Дорожная карта (по этапам)

## Этап 0 — Стабилизация (1–2 дня)

### Задачи
1. Восстановить сборку frontend:
   - добавить `frontend/lib/utils.ts`;
   - добавить `frontend/lib/api.ts` с типами и единым API-клиентом;
   - проверить `npm run build` и исправить импорты.
2. Проверить совместимость Docker-сервисов и имен в README/Makefile.
3. Зафиксировать минимальный smoke-check:
   - backend стартует;
   - frontend билдится;
   - базовые endpoints отвечают.

### Результат этапа
- Репозиторий снова собирается «из коробки».

---

## Этап 1 — Безопасность и конфигурация (2–3 дня)

### Задачи
1. Вынести все секреты в env:
   - JWT secret;
   - CORS origins;
   - DB credentials;
   - параметры PgAdmin.
2. Убрать небезопасные default-значения из runtime-конфига.
3. Добавить централизованный модуль security-конфигурации:
   - срок жизни токена;
   - алгоритм подписи;
   - refresh strategy (если нужен).
4. Усилить admin auth:
   - серверная проверка JWT на защищенных маршрутах;
   - убрать доверие к `adminLoggedIn` как единственному источнику авторизации.

### Результат этапа
- Нет секретов в исходниках, авторизация опирается на серверную валидацию.

---

## Этап 2 — Архитектурное выравнивание backend (3–5 дней)

### Задачи
1. Привести DI в соответствие с реализацией репозиториев:
   - либо полноценный DB-backed `AdminRepositoryImpl`,
   - либо корректный отдельный in-memory режим для dev/demo.
2. Исправить repository-методы поиска по `id`:
   - использовать фильтрацию по первичному ключу вместо индекса в списке.
3. Исправить проблемные места в моделях и схемах:
   - опечатки `__tablename__`;
   - согласованность локализованных полей;
   - пересмотр `unique=True` для контентных полей.
4. Подготовить и прогнать миграции Alembic после правок модели.

### Результат этапа
- Чистая, предсказуемая data-access-логика без скрытых архитектурных «разрывов».

---

## Этап 3 — Качество кода и тесты (3–4 дня)

### Задачи
1. Добавить backend тесты:
   - unit-тесты use-cases;
   - repository-тесты;
   - API-тесты (auth, page endpoints, admin endpoints).
2. Добавить frontend проверки:
   - линт и typecheck;
   - smoke-тесты ключевых экранов (главная, admin login).
3. Настроить минимальные quality-gates:
   - `pytest`;
   - `npm run lint`;
   - `npm run build`.

### Результат этапа
- Повышенная уверенность в регрессиях, воспроизводимое качество релизов.

---

## Этап 4 — CI/CD и эксплуатация (2–3 дня)

### Задачи
1. Настроить CI pipeline (например, GitHub Actions):
   - backend lint/test;
   - frontend lint/build;
   - опционально docker build.
2. Добавить шаблон релизного процесса:
   - changelog;
   - версия;
   - миграции.
3. Добавить runtime observability:
   - структурированное логирование;
   - healthcheck endpoints;
   - базовые алерты по доступности.

### Результат этапа
- Автоматизированная проверка и более предсказуемый деплой.

---

## Этап 5 — UX и продуктовые улучшения (опционально, 1+ неделя)

### Задачи
1. Вынести API-слой frontend в typed client + error normalization.
2. Улучшить админ-панель:
   - явные состояния загрузки/ошибок;
   - optimistic updates;
   - аудит-лог изменений.
3. Оптимизировать контентные секции:
   - кэширование;
   - lazy loading для тяжелых блоков.

### Результат этапа
- Лучший UX и управляемость продукта.

## 4. Приоритеты (что делать в первую очередь)

1. **P0:** вернуть успешную сборку frontend и выровнять документацию запуска.
2. **P0:** устранить security-риски (секреты, JWT, CORS, admin auth).
3. **P1:** исправить backend-архитектурные рассогласования и модели БД.
4. **P1:** нарастить покрытие тестами.
5. **P2:** CI/CD, observability, UX-оптимизация.

## 5. Definition of Done (на уровне модернизации)

- `backend`: тесты проходят, критические endpoints покрыты.
- `frontend`: линт и production build проходят стабильно.
- Нет секретов/паролей в коде.
- README и docker-конфигурация синхронизированы.
- CI блокирует merge при падении проверок.

## 6. Риски и меры снижения

- **Риск:** ломка API-контрактов при рефакторинге.
  - **Мера:** сначала зафиксировать контракты через схемы и тесты.
- **Риск:** миграции повредят существующие данные.
  - **Мера:** pre-migration backup + dry-run на staging.
- **Риск:** затяжной рефакторинг без value.
  - **Мера:** короткие итерации по этапам, каждый этап дает измеримый результат.

## 7. Метрики успеха

- Успешность CI (target: >95% green builds).
- Время восстановления после падения сборки.
- Покрытие backend-тестами (минимум 60% на критических модулях).
- Количество security-находок (должно снижаться до 0 критических).
